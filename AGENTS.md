# sessioncheck – Agent Instructions

## Project Overview

**sessioncheck** is a lightweight R package that provides tools for
checking whether an R session is in a “clean” state. It is intended to
be called at the top of a script as a drop-in replacement for the common
(but misleading) idiom of `rm(list = ls())`. Rather than silently wiping
the environment, it surfaces problems so the user can make informed
decisions.

## Naming & Code Conventions

- **Private functions** are prefixed with `.` (e.g.,
  `.get_package_status()`, `.validate_action()`).
- **Public functions** are exported and have no prefix.
- All parameters use `snake_case`.
- S3 class names follow the pattern `sessioncheck_<noun>` (e.g.,
  `sessioncheck_status`, `sessioncheck_sessioncheck`).
- Internal helpers live in `internal.R`; package-level utilities and
  `.onLoad` live in `utils.R`.

## Public API

The eight exported check functions share a consistent interface:

``` r
check_*(action = "warn", allow_*/required_*/max_* = NULL)
```

- `action`: one of `"error"`, `"warn"`, `"message"`, `"none"`.
- The second argument is check-specific (an allowlist, a required-values
  list, or a numeric threshold).
- All return an invisible S3 status object.

[`sessioncheck()`](https://sessioncheck.djnavarro.net/reference/sessioncheck.md)
is the top-level orchestrator; it delegates to individual `check_*`
functions and accepts their arguments via `...`.

Configuration also flows from `options(sessioncheck = list(...))`, which
is merged with explicit arguments. The precedence is:

    explicit arg > options(sessioncheck = ...) > function default > hard-coded allowlist

## S3 Classes

Two classes are defined in `class.R`:

| Class | Description |
|----|----|
| `sessioncheck_status` | Result of a single check. Contains a named logical vector (`status`) and a `type` string. `TRUE` means a problem was detected. |
| `sessioncheck_sessioncheck` | Composite result from [`sessioncheck()`](https://sessioncheck.djnavarro.net/reference/sessioncheck.md). A list of `sessioncheck_status` objects. |

Both classes have `format`, `print`, and `as.data.frame` methods.

## Hard-coded Allowlists (do not change without care)

Several things are always allowed regardless of user configuration:

- Packages with `"base"` priority on the search path (they cannot be
  safely detached).
- The `sessioncheck` namespace itself (always loaded while checks run).
- Dot-prefixed objects in `.GlobalEnv` (e.g., `.Random.seed`).
- IDE-injected environments (`"tools:rstudio"`, `"tools:positron"`,
  `"Autoloads"`).

## Do Not Touch Without Explicit Instruction

### Auto-generated files — never edit directly

- **`man/*.Rd`** — generated by roxygen2 via `devtools::document()`.
  Edit the roxygen2 comments in the `R/` source files instead.
- **`NAMESPACE`** — also generated by roxygen2. Never edit by hand.
- **`tests/testthat/_snaps/`** — snapshot files are generated by
  testthat. Only update them after deliberately reviewing the diff with
  [`testthat::snapshot_review()`](https://testthat.r-lib.org/reference/snapshot_accept.html);
  do not edit the `.md` files directly.

### Public API — no changes without explicit approval

- **Exported function names** — renaming or removing any exported
  function is a breaking change for users.
- **Function signatures** — do not add, remove, rename, or reorder
  parameters of any `check_*()` function or
  [`sessioncheck()`](https://sessioncheck.djnavarro.net/reference/sessioncheck.md).
  Default values count: changing `action = "warn"` to any other default
  is a breaking change.
- **S3 class structure** — do not change the fields of
  `sessioncheck_status` or `sessioncheck_sessioncheck`, and do not add
  or remove S3 methods, without explicit instruction. These classes are
  part of the user-visible contract.

### Dependencies — the zero-import rule is strict

- Do not add any package to `Imports` in `DESCRIPTION` for any reason.
  If a task seems to require an external package, stop and raise it with
  the user instead.
- `Suggests` additions (e.g., for a new test helper) are acceptable but
  should still be flagged before adding.

### Hard-coded allowlists — safety-critical

- Do not modify the logic in `internal.R` that unconditionally allows
  base-priority packages, the `sessioncheck` namespace, dot-prefixed
  globals, or IDE-injected environments. These rules exist to prevent
  the package from producing false positives or breaking in normal R
  sessions. Changes here require explicit discussion.

### Opportunistic refactoring — don’t

- Do not refactor, reformat, or reorganize code in files beyond the
  scope of the task at hand, even if it looks like it could be cleaner.
  Keeping diffs small and focused makes review much easier.

## Development Workflow

This package uses standard R package development tools:

``` r

# Load package for interactive development
devtools::load_all()

# Run full test suite
devtools::test()

# Run a specific test file
testthat::test_file("tests/testthat/test-api.R")

# Check the package (equivalent of R CMD check)
devtools::check()

# Regenerate documentation from roxygen2 comments
devtools::document()

# Build and preview pkgdown site
pkgdown::build_site()
```

Snapshot tests live in `tests/testthat/_snaps/`. Update them with
[`testthat::snapshot_review()`](https://testthat.r-lib.org/reference/snapshot_accept.html)
or `devtools::test()` after confirming changes are intentional.

## Testing Patterns

- **Framework**: testthat edition 3.
- **Mocking**:
  [`testthat::local_mocked_bindings()`](https://testthat.r-lib.org/reference/local_mocked_bindings.html)
  is used to isolate checks from actual session state. Use the same
  approach when adding tests for new checks.
- New public functions should have tests in the appropriate `test-*.R`
  file covering: normal operation, edge cases, and the effect of each
  `action` value.
- Validators are tested separately in `test-validators.R`.

## Design Principles to Preserve

1.  **Zero external dependencies.** Do not add any packages to
    `Imports`. The package must work with base R alone.
2.  **Non-destructive by default.** Checks warn rather than error by
    default. The user always retains control.
3.  **Transparent allowlisting.** Legitimate session state can always be
    whitelisted; nothing is silently ignored.
4.  **Consistent function signatures.** New check functions should
    follow the existing `check_*(action, allow_*/required_*/max_*)`
    pattern and return an invisible `sessioncheck_status` object created
    with `new_status()`.
5.  **Early validation.** Argument validation happens at the top of each
    function via `.validate_*()` helpers before any session inspection
    occurs.
